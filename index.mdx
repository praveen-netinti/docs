# Quick Prep — 7 Answers That Cover 80%

> Low on time? Learn these 7. Almost every assignment question is one of these themes asked a different way. Each answer flexes — start with the part they asked about, then let the rest follow.
>
> **Universal shape for any answer:** Answer first → why → the trade-off → tie back to real work if it fits. Short sentences. Contractions. Never defensive — a cut feature was a decision, not a mistake.

---

## 1. The Overview (your opener for almost anything)

> I built a tracing dashboard for LLM apps — you can look at traces, dig into a single trace and its spans, track issues, and see the analytics like cost, tokens, and error rates. I styled it to feel like Linear: quiet, dense, fast.
>
> The way I structured it, each layer does one job. Raw data sits in JSON. A layer of plain functions turns that into what the UI needs. State lives in React Context. Then there's a library of generic UI components, and the pages put it all together. Data flows one direction, so if I swap the JSON for a real API later, I only touch the data layer — the components don't care where the data came from.
>
> I ran it on static mock data on purpose. A backend would've eaten the time and shown nothing about my front-end work, which is the point of the role. So I spent the time on the component library and the interactions instead.

**Covers:** Tell me about the assignment · Why this architecture · Walk me through the folder structure · Why static JSON · How would another dev understand this · How do you separate logic from UI.

---

## 2. Priorities & Trade-offs (the most reused answer)

> My priorities were, in order: the component library, because everything sits on it; then interaction quality — keyboard nav, the resizable panel, drag and drop, the command menu; then the design polish.
>
> The trade-offs were no tests, no runtime validation, and no real backend. That was a time call. With fixed mock data, those add less value than getting the feel right. So it was a deliberate choice to spend the time where it shows.
>
> The thing is, I know exactly where those gaps are, and I can walk through what I'd add first. For a take-home I think it was the right call. For a real product, every one of those flips to "yes."

**Covers:** What were your priorities · What trade-offs did you make · What corners did you cut · What compromises did you make · What did you intentionally not do.

## 3. Tests (very likely — say this with confidence, not apology)

> That was a time call. With fixed mock data, the riskiest part is the interactions and how things render, not the logic. So I spent the time getting the interactions actually right by hand — keyboard nav, focus, drag and drop — instead of writing shallow tests that pass but don't catch those.
>
> I also kept the logic in plain functions, separate from the UI, so tests drop in cleanly later. The first thing I'd test is those functions — flattening traces, computing stats — they're fast and a bug there shows up everywhere. Then component tests with React Testing Library, and Playwright for the real flows like creating an issue. All of it running in CI on every change.
>
> One thing I'm particular about — I only mock the network, never my own hooks or stores. Otherwise you're just testing the mock.

**Covers:** Why no tests · What would you test first · Unit vs integration · Why Vitest/RTL/Playwright · How would CI look · Mocking · Accessibility/visual testing (just add: "I'd add automated a11y checks and screenshot tests too").

---

## 4. Validation / Zod (very likely)

> Right now all the data is JSON I wrote myself, and TypeScript already covers it. A schema would be checking data that can't really be wrong. So it would've been effort for no real gain here.
>
> The one real gap is where I read issues back from localStorage — that's a parse with no check, so a stale or hand-edited value would slip through. That's exactly where I'd add validation first.
>
> More generally, validation belongs anywhere data comes from outside my code — API responses, localStorage, URL params. TypeScript only checks while you write code; it's gone at runtime. A schema checks the real data while the app runs. That's why TypeScript alone isn't enough at the edges.

**Covers:** Why no Zod · Where would you add it · Why validate API responses · Validate forms / URL params / localStorage · Runtime vs compile-time · Why TypeScript isn't enough.

## 5. Performance & Scale (one answer for the whole cluster)

> The honest answer is it'd slow down with a lot of data right now, because the table renders every row. That's the first thing I'd fix.
>
> The fix is virtualization — only render the rows that are actually on screen and swap them as you scroll. The table's built on a headless table library, so I can add that without touching the sorting or selection logic. Then I'd stop loading everything at once and page the data from the server, and move the heavy read pages to render on the server so the browser ships less.
>
> But I'd measure before optimizing — Lighthouse for load, the React profiler for re-renders — and fix the real hot spots instead of guessing. At MyBizz I cut initial load roughly in half, mostly by splitting code per page and loading heavy parts only when they're needed. So I've done this for real.

**Covers:** What happens with 100k/1M traces · How would you optimize rendering · Where are the bottlenecks · How would you measure performance · Virtualization · Code splitting · Lazy loading · Bundle size · How would you scale this · What performance work have you done.

## 6. State, Components & Data flow (the "why these choices" cluster)

> I chose React Context because the shared state is small — the issues list, sidebar, shortcuts, settings. Context handles that without adding another library, and it keeps the project easy to understand because it's just React. For the one thing that changes fast, notifications, I used a small external store instead, because Context would re-render too much there.
>
> If the shared state grew, I'd reach for Zustand before Redux — it's lighter and lets components subscribe to just the slice they need. Redux would've been a lot of boilerplate for very little gain here.
>
> For the components, I built my own on top of accessible primitives, so the behavior that's easy to get wrong is handled and the look and the API are mine. My main rule is composition over a pile of boolean flags — instead of a button with leftIcon, rightIcon, loading, you give it children. Fewer props, more flexible. That's the same thinking as the UI Kit I shipped across six products at MyBizz.

**Covers:** Why Context · Why no Redux/Zustand · When would you move to Zustand · What belongs in Context · Why build your own components · How reusable are they · Composition · Component API design · Why not more props.

---

## 7. Improvements, Weakness & Production (self-critique cluster)

> If I had another week, I'd start with the foundation — set up CI, add tests starting with the plain logic functions, and add validation where data comes in. Then connect a real backend, move the read-heavy pages to the server, and virtualize the table. I'd do it in stages, foundation before features.
>
> The weakest part right now is the issues data flow — it reads from localStorage on every change instead of trusting React state. It works, but it's doing extra work. That's the first thing I'd refactor.
>
> The one decision I'd reconsider is making everything a client component. For a static demo it was fast, but the read-heavy pages really should render on the server. If this became a real product, that plus real data with validation and tests in CI is what I'd change first.

**Covers:** What would you improve with more time · Biggest weakness · What would you refactor · Which decision would you reconsider · If you started over · If this became production what changes first · What would production require · Technical debt · Code smells · CI/CD, monitoring, error tracking (add: "plus error tracking and monitoring so I catch problems in the field").

## If you only memorize three

**3 (tests)**, **4 (validation)**, and **2 (trade-offs)** — those are the ones they're most likely to grill, and they're the ones where sounding calm instead of apologetic makes the biggest difference.

## Two lines to never forget

- "That was a deliberate decision — with fixed mock data, I spent the time where it shows: the components and the interactions."
- "I know exactly where the gaps are. For a take-home it was the right call; for a real product every one flips to yes."

## If you get something you didn't prep

> "I haven't done that directly, but here's how I'd approach it..." — then reason out loud. Thinking clearly beats reciting. They're testing how you think, not what you memorized.
